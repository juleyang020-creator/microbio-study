# 微生物学习软件 — 设计文档

- 日期：2026-06-03
- 状态：待用户确认
- 形态：本地静态网页应用（双击 `index.html` 打开，免安装、可离线）

## 1. 概述

一个面向"系统学习与查阅"的微生物学知识库软件。第一阶段（本设计）目标是**搭好框架**：把信息架构、数据模型、界面布局、三大模块的分类骨架做扎实，并放入少量占位条目；真实知识内容后续逐步填充。

三大模块：

1. **微生物分类**
2. **抗生素分类及作用原理**
3. **耐药因素分类及原理**

三者在数据上互相打通，可跨模块跳转。

## 2. 目标与非目标

**本期目标（v1 框架）**

- 三大模块的导航 + 分类树骨架
- 统一的数据模型，支持跨模块关联（声明一次、两边自动显示）
- 主界面：顶部模块切换 + 全局搜索；左侧分类树；右侧条目详情 + 关联跳转
- 每个模块放几个占位条目，验证结构跑通
- 数据自检（id 重复 / 悬空关联 / 未匹配分类）的控制台提醒

**非目标（明确推迟到后续"细分场景"）**

- 自测 / 刷题 / 闪卡等主动记忆功能（数据结构为其预留可能，但本期不做）
- 软件内的可视化内容编辑器（本期用数据文件维护）
- 账号、云同步、多人协作
- 更细的"关系类型"（如"能治疗 / 被对抗"）——本期用通用关联，需要时再加

## 3. 技术路线

- **纯静态网页，免构建**：HTML + CSS + 原生 JavaScript。
- 双击 `index.html` 即可在浏览器打开（`file://` 协议）。
- 因此数据文件以 `<script>` 方式加载并挂到全局对象（**不使用 `fetch`/ES 模块**，避免 `file://` 下的跨域限制）。
- 无外部依赖、不联网、不收集任何数据。
- 数据与界面分离：界面逻辑在 `js/`，内容在 `data/`。

## 4. 信息架构与分类骨架

抗生素、耐药两个模块均**按"机制"组织**，以贴合"作用原理 / 耐药原理"的学习目标，并便于机制对机制地对照。

### ① 微生物分类
- 细菌：革兰氏阳性球菌 / 革兰氏阴性杆菌 / 革兰氏阳性杆菌 / 分枝杆菌 / 特殊类（支原体·衣原体·螺旋体…）
- 病毒：DNA 病毒 / RNA 病毒
- 真菌
- 寄生虫（原虫 / 蠕虫）

### ② 抗生素（按作用机制）
- 抑制细胞壁合成：β-内酰胺类（青霉素 / 头孢 / 碳青霉烯）/ 糖肽类
- 抑制蛋白质合成：氨基糖苷 / 大环内酯 / 四环素 / 林可酰胺
- 抑制核酸合成：喹诺酮 / 利福平 / 甲硝唑
- 抑制叶酸代谢：磺胺类
- 破坏细胞膜：多黏菌素 / 达托霉素

### ③ 耐药因素（按机制）
- 产生灭活酶：β-内酰胺酶（ESBL / 碳青霉烯酶）/ 氨基糖苷修饰酶
- 靶位改变：PBP2a（MRSA/mecA）/ 核糖体甲基化 / VanA
- 主动外排泵
- 膜通透性下降（孔蛋白缺失）
- 旁路代谢 / 生物膜
- 备注："获得方式（固有 / 突变 / 质粒转移）"可作为条目内的小节

> 以上为起始骨架，可随时改名、增减、调整顺序与层级。

## 5. 数据模型

### 条目（三大模块共用同一结构）

```js
{
  id: "staph-aureus",        // 全局唯一编号（英文/拼音），程序用
  名称: "金黄色葡萄球菌",
  拉丁名: "Staphylococcus aureus",  // 可选
  类别: "革兰氏阳性球菌",     // 对应所属模块分类树的某个叶子
  小节: [                    // 详情正文，可自由增删；每个小节 = 一张字段卡片
    { 标题: "形态与染色", 正文: "（待填）" },
    { 标题: "致病性",     正文: "（待填）" }
  ],
  关联: ["oxacillin", "mrsa-meca"]   // 相关条目 id，可跨模块
}
```

### 分类树（`data/categories.js`）

```js
window.DB.categories = {
  microbes:    [ { 名称: "细菌", 子类: [ { 名称: "革兰氏阳性球菌" }, … ] }, … ],
  antibiotics: [ { 名称: "抑制细胞壁合成", 子类: [ … ] }, … ],
  resistance:  [ { 名称: "产生灭活酶", 子类: [ … ] }, … ]
}
```

- 本期采用**两级树**（大类 → 子类），条目的 `类别` 匹配某个子类名称。可后续加深层级。
- 显式定义分类树（而非从条目反推），以便空分类也能在骨架中显示、并控制顺序。

### 关联解析（声明一次，两边显示）

- 应用启动时建立全局索引：`id → { 条目, 所属模块 }`。
- 渲染某条目时：
  - **正向**：遍历其 `关联`，解析出名称与所属模块，显示为可点击标签（颜色区分模块）。
  - **反向**：扫描全部条目，凡 `关联` 中包含当前条目 id 的，也一并显示（去重）。
- 因此每条关联只需在一侧声明，双方页面都会出现。

## 6. 界面设计

- **顶部栏**：应用名 + 三大模块切换标签 + 全局搜索框。
- **左侧栏**：当前模块的分类树，可展开/折叠到具体条目。
- **右侧主区**：选中条目的详情——标题 + 类别徽章（+ 拉丁名），下接若干"小节卡片"，底部为"🔗 关联"标签区（点击跳转到对应条目）。
- **搜索**：全局匹配 名称 / 拉丁名 / 小节正文，结果按模块分组列出。
- **路由**：使用 URL hash，形如 `#/<module>/<id>`，便于前进/后退与刷新保持位置。
- 视觉风格：清爽的医学/临床观感（青绿 + 中性灰白为主），信息密度适中。

## 8. 内容维护

- 内容由 Claude 在后续会话中增改（用户为使用者，不负责日常维护）。

- `data/*.js` 顶部统一形如 `window.DB = window.DB || {}; window.DB.microbes = [ … ]`。

## 9. 健壮性与错误处理

- `关联` 指向不存在的 id：该标签显示为灰色并标"?"，不报错。
- `类别` 未匹配任何分类树叶子：条目归入"未分类"，仍可查看。
- 数据为空或某模块无条目：显示友好的空状态提示。
- 纯本地运行，无网络请求、无数据收集。

## 10. 测试策略

- **数据自检（`validate.js`）**：启动时检查并在控制台 `console.warn` 提示：
  - 重复的 `id`
  - 悬空的 `关联`（指向不存在的条目）
  - 未匹配分类树的 `类别`
  - 自检为非致命，仅提醒，不阻断页面。
- **手动验收清单**：三模块切换正常；分类树展开/选中正常；详情渲染（含小节、徽章、拉丁名）正常；关联正/反向跳转正常；搜索命中并跳转正常；占位条目齐全；从 `file://` 双击打开可用。
